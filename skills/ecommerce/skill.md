# B2C eCommerce (Storefront, Orders, & Refunds)

Use this skill when developing or modifying features related to:
- B2C **Storefront Catalog & Discovery** (public products feed, cached listing)
- **Cart & Line Reservations** (adding items, TTL-based stock locks)
- **Idempotent Checkout Transactions** (cart promotion, shopper accounts, Stripe payments, down-stream SO/Invoice posting)
- **Order Fulfillment** (shipping state progressions, courier/tracking links)
- **Admin Orders & Refunds Manager** (full and partial returns, restock movements, FMS reversed journal postings)

## Module surface (sidebar)

```
eCommerce (admin)
├── Orders                 — grid list of ecom orders; paid/fulfilled status filter chips
├── Refunds                — returns inbox; lists approved/rejected/requested lines
├── Products (curation)    — product catalog visibility toggle (ecom featured, hide/publish)
├── Customers (shoppers)   — shopper directory with total spend and orders metrics
└── Settings               — payment gateway credentials (masked) + storefront branding settings
```

> [!NOTE]
> Public storefront routes reside entirely under `/shop` URL structures (e.g. `pages/shop/index.vue`, `pages/shop/cart.vue`, `pages/shop/checkout.vue`) and consume public endpoints (`/api/v1/shop/...`). The admin-facing panels reside under `/ecommerce/...` paths (`pages/ecommerce/...`).

## Core Developer Workflows

### 1. Storefront Discovery & Stock Reservation (INV-RESERVE)
* Consumes the `INV-STOREFRONT` cached product endpoints to showcase variants, attributes, and pricing to public shoppers.
* Upon adding a product/variant to the cart, the frontend triggers `POST /api/v1/shop/cart` which invokes `CartService::addItem()`.
* **Reservation Lock:** The service calls `InventoryService::reserve(variant, quantity, ttl=15m)`. The item is locked for checkout; if the shopper fails to check out within 15 minutes, a cron daemon releases the reservation automatically.

### 2. Shopper Account Management & Guest Checkout
* B2C shoppers are treated as distinct entities from B2B customers. They authenticate against `EcomCustomer` via Laravel Passport password grants under the `shopper` role.
* Carts are bound to `session_token` for guest checkouts, or to the logged-in `ecom_customer_id`.
* Upon a guest logging in, the frontend sends a merge request to the backend `POST /api/v1/shop/cart/merge` which copies all guest cart lines into the shopper's active database cart.

### 3. Idempotent Checkout & Payment Confirmation
* **Initiation:** When proceeding to checkout, the shopper triggers `POST /api/v1/shop/checkout` with a unique `client_uuid` generated by the frontend. The backend creates an `EcomOrder` with status `pending_payment` and returns a payment gateway client secret (e.g., Stripe test token).
* **Idempotency Guarantee:** If the same `client_uuid` is supplied again, the `CheckoutService::initiate` method short-circuits and returns the existing order payload without creating duplicates.
* **Confirmation (Transaction):** Once Stripe successfully processes the payment, it calls `POST /api/v1/shop/checkout/confirm` (or Stripe Webhook). `CheckoutService::confirm` processes this **within a single database transaction**:
  1. Validates the payment signature / transaction token.
  2. Creates a matching central-db `ecom_payments` log.
  3. Commits the inventory reservation by invoking `StockMovement::create` (type=`ecom_sale`).
  4. Generates a downstream Sales `Order` and `Invoice` to log standard Accounts Receivable (AR) using `InvoiceService::createFromOrder`.
  5. Updates `EcomOrder` status to `paid`.

### 4. Fulfillment & Shipping Updates
* Admin staff view paid orders in `pages/ecommerce/orders/[id].vue`.
* Marking an order shipped invokes `FulfillmentService::ship(order, carrier, tracking_number)`.
* Transitions order from `paid` -> `shipped`, dispatches `EcomOrderShipped` event to trigger client email confirmations.

### 5. Admin-Initiated Full & Partial Refunds
* B2C refunds are managed exclusively by administrators in Phase 1.
* Initiating a return creates an `EcomRefund` with lines indicating returned item counts.
* **Transaction Rollback:** `RefundService::approve` runs within a single txn:
  1. Dispatches payment refund API call to the gateway.
  2. Updates inventory stock ledger with `StockMovement` (type=`ecom_restock`).
  3. Posts a reversing entry to the General Ledger via `AccountingService::postEntry` (or Credit Note if enabled).
  4. Transitions `EcomRefund` to `completed`.

## Critical Rules

### 1. Multi-Tenant Query Scoping (P0)
Always apply the `BelongsToTenant` trait on all B2C entities: `EcomCustomer`, `EcomAddress`, `EcomCart`, `EcomCartItem`, `EcomOrder`, `EcomOrderItem`, `EcomPayment`, `EcomRefund`, and `EcomRefundItem`. Bypassing tenant filters is a P0 security leak.

### 2. Double-Charge Prevention & Transaction Boundaries (P0)
- All payment callbacks and webhook settlements must be guarded by a unique `client_uuid` or `transaction_id` unique constraint check.
- Multi-table writes (decr stock, create SO, write ledger, update order status) MUST be wrapped in a `DB::transaction`. A gateway success must never exist without an inventory deduction or invoice log in the system.

### 3. Shopper Contact & Data Encryption (P0)
- Shopper data (`ecom_customers.phone` and `ecom_addresses.phone`) MUST be encrypted in the database. Use Eloquent `'encrypted'` casts.
- Password hashes must rely on standard bcrypt using Laravel Eloquent default casting rules.

### 4. JSON Serialization Contracts (P0)
Always return `JsonResource` objects directly from your controllers to keep structural integrity, preventing raw array wraps from exposing internal sentinel values like `MissingValue`.

## Troubleshooting

- **`Duplicate payment webhook processing error`**: Webhook processed after a shopper was already redirected to the thank-you page. Guard against processing by checking if `EcomOrder.status !== 'pending_payment'` before running confirmation logic.
- **`Stock reservation expiry mismatch`**: Reserved items appear out of stock but cart TTL is still active. Ensure `CartService::addItem` checks both `on_hand` stock and any unexpired reservations.
- **`AR ledger out-of-balance on refund`**: A partial refund calculated lines incorrectly. Ensure the reversing journal dynamically matches the exact sub-total of items returned plus prorated taxes.
